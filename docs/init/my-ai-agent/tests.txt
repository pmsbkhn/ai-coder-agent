PHẦN 1: ARCHITECTURAL FITNESS FUNCTIONS (Kiểm soát cấu trúc)Các bài test này (viết bằng Python sử dụng thư viện kiểm định cấu trúc hoặc bóc tách AST) nhằm ép buộc AI khi phát triển lõi Orchestrator không được phép phá vỡ tính đóng gói Hexagonal.TC-ARCH-01 [Strict Domain Isolation]: Quét toàn bộ package domain/ của AI Agent. Thử nghiệm import các thư viện bên ngoài như openai, anthropic, chromadb, hoặc neo4j -> Hệ thống phải chặn đứng và báo lỗi vi phạm kiến trúc. Tầng Domain chỉ được phép chứa các cấu trúc Python thuần và định nghĩa kiểu dữ liệu.TC-ARCH-02 [Port Enforcement]: Trái tim điều phối Orchestrator không được phép gọi trực tiếp đến client của cơ sở dữ liệu hoặc các hàm SDK của MCP. Mọi hành vi đọc/ghi bộ nhớ, gọi LLM, hay ra lệnh cho Tool bắt buộc phải đi qua các Output Ports độc lập (PlannerPort, MemoryPort, MCPGatewayPort).TC-ARCH-03 [Append-Only Enforcer]: Quét module MemoryAdapter. Bất kỳ câu lệnh SQL hoặc logic xử lý tệp nào chứa từ khóa UPDATE hoặc DELETE hướng tới bảng ghi vết ExecutionTrace (Log DB) đều phải bị phủ quyết. Chỉ chấp nhận phương thức INSERT hoặc ghi nối dòng (append).PHẦN 2: DOMAIN INVARIANT TEST CASES (Kiểm soát nghiệp vụ lõi)Tầng này kiểm tra các bất biến trạng thái (Invariants) nằm bên trong vòng lặp điều khiển (Control Loop) và thực thể quản lý phiên làm việc (AgentSession/CheckoutSaga tương đương của Agent). Mọi bài test sử dụng kỹ thuật Fake/Mock các Port đầu ra.1. Luồng Lập kế hoạch & Khử trùng lặp (Planning & Idempotency)TC-CORE-01 [Linear Saga State Transition]: Khởi tạo một phiên xử lý yêu cầu (AgentSession) ở trạng thái INIT. Gọi lệnh yêu cầu Coder sửa code khi trạng thái phiên chưa chuyển dịch qua bước PLANNING (chưa có danh sách task con) -> Hệ thống bắt buộc phải ném lỗi InvalidStateTransitionException.TC-CORE-02 [Idempotency Distributed Lock]: Giả lập người dùng gửi trùng một RequirementPrompt hai lần liên tiếp với cùng một mã IdempotencyKey. Tại lần gọi thứ hai, mock MemoryPort trả về kết quả lock đang tồn tại (LockExists) -> Assert hệ thống trả ngay về trạng thái "Task is processing", chặn đứng luồng gọi sang PlannerPort để không bị tốn token vô ích.2. Chiến lược Đọc mã nguồn (Interactive Lazy-Loading RAG)TC-CORE-03 [Skeleton-First Rule]: Khi nhận yêu cầu sửa code từ người dùng, lệnh gọi đầu tiên của Orchestrator qua MCPGatewayPort hướng tới Code-reader Server bắt buộc phải là lệnh yêu cầu trích xuất Repo Map (Skeleton). Assert payload gửi đi không được chứa yêu cầu đọc toàn bộ file chi tiết (Nói không với Batch tĩnh).TC-CORE-04 [Zoom-In Request Validation]: Sau khi nhận được Repo Map nén, Orchestrator dựa vào phân tích từ PlannerPort để đưa ra Tool Call tiếp theo. Assert Tool Call này phải có cấu trúc tường minh: Chỉ định đích danh đường dẫn file (ví dụ: path/to/File.java) và tên Class/Method cụ thể thông qua cú pháp của GritQL hoặc Tree-sitter.3. Vòng lặp Tự sửa lỗi (Self-Healing Loop & Circuit Breaker)TC-CORE-05 [Self-Healing Prompt Augmentation]: Giả lập Verifier trả về kết quả FAIL kèm stack trace lỗi biên dịch của Maven ở vòng lặp thứ nhất. Ở vòng lặp thứ hai, assert cấu trúc Prompt gửi sang Coder bắt buộc phải được tự động nối thêm dữ liệu vết lỗi (ExecutionTrace) lấy từ MemoryPort để Coder biết đường tự sửa.TC-CORE-06 [Circuit Breaker Trigger]: Cấu hình số lần thử tối đa $N = 3$. Giả lập Verifier liên tục trả về kết quả FAIL qua cả 3 vòng lặp. Assert tại vòng lặp thứ 4, Orchestrator phải tự động kích hoạt Circuit Breaker, dừng ngay lập tức việc gọi Coder, và ghi nhận trạng thái phiên là HEALING_FAILED để trả kết quả về cho người dùng.PHẦN 3: INTEGRATION & SANDBOX TESTS (Kiểm thử hạ tầng & Bảo mật)Tầng này kiểm tra tính đúng đắn khi Orchestrator kết nối thực tế với MCP Gateway và các cụm Database mồi trên Mac M4 local (Docker).1. Zero Trust & Security GatewayTC-INT-01 [Secure Gate Interception]: Orchestrator phát một Tool Call yêu cầu thực thi lệnh triển khai (Deploy CI/CD). Yêu cầu này đi qua SecureGatePort -> Assert hạ tầng mạng phải chặn đứng luồng tự động, đệ trình yêu cầu lên Guard Gate và treo thread để chờ tín hiệu ApprovalCallbackPort từ con người (Human-in-the-loop).TC-INT-02 [Human Rejection Recovery]: Tiếp tục từ TC-INT-01, giả lập người dùng bấm Từ chối (Reject) trên giao diện Guard Gate -> Assert Orchestrator nhận được callback hủy bỏ, không gửi lệnh xuống CICD Server, giải phóng tài nguyên an toàn và ghi log DEPLOY_REJECTED.2. Đa cơ sở dữ liệu (Multi-model DB Storage)TC-INT-03 [Log DB Overwrite Prevention]: Thực hiện cấu hình Row-Level Security (RLS) trên bảng agent_execution_logs của PostgreSQL. Cho MemoryAdapter chạy một lệnh UPDATE hoặc DELETE giả lập can thiệp vào một log ID đã có -> Assert Database phải quăng lỗi Permission Denied / Access Denied, đảm bảo tính bất biến tuyệt đối của lịch sử.TC-INT-04 [Graph DB Dependency Mapping]: Sau khi Code-reader Server hoàn tất quét dự án bằng Tree-sitter, dữ liệu cấu trúc được nạp vào Graph DB (Neo4j). Thực hiện một câu lệnh Cypher Query tra cứu mối quan hệ của Class X -> Assert kết quả phải trả về đầy đủ danh sách các cạnh (edges) trỏ đến các Interface mà Class X đang thực thi.3. MCP JSON-RPC Protocol CompatibilityTC-INT-05 [MCP Server Pluggability]: Tắt (stop) dịch vụ Code Reader Server nhưng vẫn giữ MCP Gateway chạy. Orchestrator gửi một Tool Call đọc mã nguồn -> Assert hệ thống nhận lại đúng mã lỗi tiêu chuẩn JSON-RPC -32601 (Method not found) từ Gateway, bọc mã lỗi này vào một DomainException và kích hoạt luồng thoát an toàn, không được để crash toàn bộ tiến trình core của Agent.
