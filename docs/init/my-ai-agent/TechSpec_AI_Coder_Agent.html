2. Chiến lược Quản lý Dữ liệu (Immutable State & Multi-model Database)

Hệ thống chia tách lớp lưu trữ thành 3 nhóm cơ sở dữ liệu chuyên biệt để phục vụ quá trình suy luận (Reasoning) và tự học (Self-healing):

  • Vector Database (RAG & Semantic Search): Sử dụng ChromaDB, Qdrant hoặc pgvector (tối ưu cho local Mac M4). Lưu trữ nội dung file, cấu trúc class và document. Cung cấp khả năng truy vấn ngữ nghĩa để tìm kiếm các file code liên quan mật thiết đến yêu cầu sửa đổi, cấp dữ liệu mồi cho Planner.
  • Graph Database (Dependency & Architecture Quanta): Sử dụng Neo4j hoặc Kùzu. Lưu trữ mạng lưới chằng chịt của mã nguồn (ví dụ: Class A extends B, gọi interface C). Đảm bảo LLM có cái nhìn toàn cảnh về cấu trúc, tránh việc sửa một module làm gãy dây chuyền (ripple effect) toàn hệ thống.
  • Time-Series / Append-Only Log (Tính Bất biến): PostgreSQL (cấu hình RLS chỉ cho phép INSERT) hoặc thư mục chứa file .jsonl. Lưu trữ toàn bộ execution trace, log lỗi Maven, và quyết định của Verifier qua từng loop. Đảm bảo thuộc tính **Chống Overwrite/Override**, giúp Agent học từ sai lầm (Self-healing) mà không bị mất dấu vết ngữ cảnh cũ.

3. Phân tích Mã nguồn bằng Interactive Lazy-Loading RAG

Nói KHÔNG với việc nhồi toàn bộ codebase vào context window (gây Lost in the Middle) hoặc cắt nhỏ ngẫu nhiên theo chunk (gây đứt gãy syntax). Hệ thống áp dụng quy trình 2 bước của các Agentic System hàng đầu:

  1. Chụp X-Quang (Skeleton / Repo Map): Sử dụng module Repo Map (dựa trên Tree-sitter và thuật toán PageRank, tương tự Aider) để sinh ra một bản đồ tóm tắt ngắn gọn. Bản đồ này chỉ chứa cây thư mục và chữ ký hàm (method signatures) mà không chứa nội dung chi tiết.
  2. Zoom-in (Retrieve theo yêu cầu): Dựa vào Repo Map, Planner tự nhận định khu vực cần can thiệp và gọi Function Calling (qua MCP Gateway) yêu cầu: "Trích xuất chi tiết nội dung Class X và Y". Các công cụ AST như Tree-sitter, Bloop hoặc GritQL đảm bảo kết quả trả về chính xác theo cấu trúc mã nguồn.

4. Thiết kế chi tiết Port & Giao diện Orchestrator

Việc áp dụng kiến trúc Model Context Protocol (MCP) giúp Orchestrator tinh gọn tối đa. Nó chỉ giao tiếp với một MCPGatewayPort duy nhất, chuyển giao việc quản lý chi tiết các công cụ (Git, Maven, Code Reader) cho lớp MCP Server.

package tech.vsf.ai.agent.application.port.in;

// 4.1 Inbound Ports: Tiếp nhận tương tác
public interface RequirementPort {
    void submitRequirement(String prompt);
}

public interface FeedbackPort {
    void processVerificationResult(VerificationResult result);
}

package tech.vsf.ai.agent.application.port.out;

// 4.2 Outbound Ports: Gọi dịch vụ phụ trợ
public interface PlannerPort {
    Plan generatePlan(String requirement, RepoMap repoMap);
}

public interface MemoryPort {
    void appendLog(ExecutionTrace trace);
    RepoMap getRepoMap();
}

public interface MCPGatewayPort {
    // Port hợp nhất gọi Tool qua JSON-RPC (MCP Protocol)
    ToolResponse executeToolCall(ToolRequest request);
}
